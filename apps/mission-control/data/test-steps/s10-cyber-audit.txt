# OWASP Top 10 Security Audit — Beauty CRM Loyalty Module

---

## A01 — Broken Access Control

### FINDING 1
**SEVERITY:** Critical  
**FILE:** `src/modules/clients/clients.controller.ts`  
**VULNERABILITY:** `GET /clients/:id` and `PATCH /clients/:id` have no location-scoping — any authenticated staff/owner JWT from *any* location can read or modify any client by UUID.

**Evidence:**
```typescript
@Get(':loyalty_client_id')
async getById(@Param('loyalty_client_id', ParseUUIDPipe) id: string) {
  return this.service.getById(id); // no location_id check
}
```

`ClientsRepository.findById` queries only `WHERE lc.id = $1` — no `location_id` filter. A staff member at Location A can enumerate clients from Location B by brute-forcing UUIDs (low-entropy due to sequential Postgres UUIDs in some configs).

**FIX:**
```typescript
// clients.controller.ts
@Get(':loyalty_client_id')
async getById(
  @Param('loyalty_client_id', ParseUUIDPipe) id: string,
  @Request() req: any,
) {
  return this.service.getById(id, req.user.location_id);
}

// clients.repository.ts
async findById(id: string, locationId: string) {
  const result = await this.pool.query(
    `SELECT lc.*, lt.name as tier_name
     FROM loyalty_clients lc
     LEFT JOIN loyalty_tiers lt ON lt.id = lc.current_tier_id
     WHERE lc.id = $1 AND lc.location_id = $2 AND lc.deleted_at IS NULL`,
    [id, locationId],
  );
  return result.rows[0] || null;
}
```

**SEVERITY:** Critical  
**FILE:** `src/modules/redemptions/redemptions.controller.ts` + `src/modules/redemptions/redemptions.service.ts`  
**VULNERABILITY:** `GET /redemptions/:code` has no authentication guard — any unauthenticated request can look up any redemption code and retrieve `loyalty_client_id`, `reward_name`, `point_cost`, and `status`.

**Evidence:**
```typescript
@Get(':code')
```

**FIX:**

```typescript
// redemptions.service.ts

  if (!redemption || redemption.location_id !== locationId)
    throw new NotFoundException('Code not found');
  // ...
}
```

### FINDING 3
**SEVERITY:** High  
**FILE:** `src/auth/roles.guard.ts`  
**VULNERABILITY:** `Roles` decorator uses `Reflect.defineMetadata` directly on `descriptor.value` instead of NestJS `SetMetadata`, causing `Reflector.get(ROLES_KEY, context.getHandler())` to always return `undefined` — all role-protected routes silently pass through with any role.

**Evidence:**
```typescript
export function Roles(...roles: string[]) {
  return (target: any, key: string, descriptor: PropertyDescriptor) => {
    Reflect.defineMetadata(ROLES_KEY, roles, descriptor.value); // wrong target
    return descriptor;
  };
}
// In guard:
const requiredRoles = this.reflector.get<string[]>(ROLES_KEY, context.getHandler());
// context.getHandler() !== descriptor.value → always undefined → guard returns true
```

**FIX:**
```typescript
import { SetMetadata } from '@nestjs/common';

export const ROLES_KEY = 'roles';
export const Roles = (...roles: string[]) => SetMetadata(ROLES_KEY, roles);

// Guard remains unchanged — Reflector.get works correctly with SetMetadata
```

## A02 — Cryptographic Failures

### FINDING 5
**SEVERITY:** High  
**FILE:** `src/common/utils/referral-code.generator.ts`  
**VULNERABILITY:** `generateReferralCode` uses modulo bias — `bytes[i] % CHARSET.length` where CHARSET has 36 characters and a byte has 256 values. 256 % 36 = 4, meaning characters at indices 0–3 appear with probability 8/256 vs 7/256 for indices 4–35, creating a measurable statistical bias that weakens brute-force resistance.

**FIX:**
```typescript
export function generateReferralCode(length = 8): string {
  const CHARSET = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789';
  // 256 / 36 = 7.11 → use rejection sampling with ceiling at 252 (7*36)
  const MAX_VALID = Math.floor(256 / CHARSET.length) * CHARSET.length; // 252
  const result: string[] = [];
  while (result.length < length) {
    const bytes = crypto.randomBytes(length * 3);
    for (const b of bytes) {
      if (b < MAX_VALID && result.length < length) {
        result.push(CHARSET[b % CHARSET.length]);
      }
    }
  }
  return result.join('');
}
```

### FINDING 6
**SEVERITY:** High  
**FILE:** `src/auth/auth.module.ts` + `src/auth/jwt.strategy.ts`  
**VULNERABILITY:** JWT secret is loaded at module instantiation via `process.env.LOYALTY_JWT_SECRET` with no validation — if the env var is absent or empty, NestJS JWT module defaults to signing with an empty string secret, allowing any attacker to forge valid tokens.

**Evidence:**
```typescript
JwtModule.register({
  secret: process.env.LOYALTY_JWT_SECRET, // undefined if not set
  signOptions: { expiresIn: '24h' },
})
```

**FIX:**
```typescript
// auth.module.ts
JwtModule.register({
  secret: (() => {
    const s = process.env.LOYALTY_JWT_SECRET;
    if (!s || s.length < 32) throw new Error('LOYALTY_JWT_SECRET must be ≥32 chars');
    return s;
  })(),
  signOptions: { expiresIn: '24h' },
})

// jwt.strategy.ts constructor
constructor() {
  const secret = process.env.LOYALTY_JWT_SECRET;
  if (!secret || secret.length < 32)
    throw new Error('LOYALTY_JWT_SECRET must be ≥32 chars');
  super({ jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
          ignoreExpiration: false, secretOrKey: secret });
}
```

### FINDING 7
**SEVERITY:** Medium  
**FILE:** `worker/src/sms/twilio.service.ts`  
**VULNERABILITY:** PWA JWT tokens are embedded in SMS message bodies (`/pwa/client?t=<token>`) and stored in `notification_logs` implicitly via the message content — tokens in SMS are logged by Twilio's platform and potentially in carrier records, exposing bearer credentials in third-party logs.

**FIX:** Use short-lived opaque link tokens that resolve server-side rather than embedding the full JWT in SMS:
```typescript
// Generate a short random token stored in Redis with TTL
const linkToken = crypto.randomBytes(16).toString('hex');
await this.redis.setEx(`pwa-link:${linkToken}`, 3600, clientId);
return `${pwaBase}/pwa/client?lt=${linkToken}`;
// PWA controller resolves linkToken → clientId → generates fresh JWT internally
```

## A03 — Injection

### FINDING 8
**SEVERITY:** Critical  
**FILE:** `src/modules/analytics/analytics.repository.ts`  
**VULNERABILITY:** `DATE_TRUNC` field parameter is passed as `$1` but PostgreSQL requires it as a string literal — the current code fails at runtime. The fix of interpolating directly into SQL creates SQL injection if granularity is not strictly validated before reaching the repository.

**Evidence:**
```typescript
const result = await this.pool.query(
  `SELECT DATE_TRUNC($1, created_at) ...`, // runtime error
  [truncFn, locationId, dateFrom, dateTo],
);
```

**FIX — safe interpolation with explicit allowlist:**
```typescript
async getPointsActivity(locationId: string, dateFrom: string, dateTo: string,
  granularity: 'day' | 'week' | 'month') {
  // Allowlist — never use user input directly
  const ALLOWED = { day: 'day', week: 'week', month: 'month' } as const;
  const trunc = ALLOWED[granularity];
  if (!trunc) throw new Error('Invalid granularity');
  
  const result = await this.pool.query(
    // Safe: trunc is from a closed enum, never user-controlled string
    `SELECT DATE_TRUNC('${trunc}', created_at) as period, ...
     FROM point_transactions
     WHERE location_id = $1 AND created_at BETWEEN $2 AND $3
     GROUP BY DATE_TRUNC('${trunc}', created_at)
     ORDER BY period ASC`,
    [locationId, dateFrom, dateTo],
  );
}
```

### FINDING 9
**SEVERITY:** High  
**FILE:** `src/database/transaction.helper.ts`  
**VULNERABILITY:** `selectForUpdate` interpolates `table` parameter directly into SQL string with no validation — callers passing attacker-controlled table names would achieve SQL injection.

**Evidence:**
```typescript
export async function selectForUpdate(client: PoolClient, table: string, id: string) {
  const result = await client.query(
    `SELECT * FROM ${table} WHERE id = $1 FOR UPDATE`, // table is not parameterized
    [id],
  );
}
```

While current callers pass string literals, the function is exported and could be called with user-influenced input in future code.

**FIX:**
```typescript
const ALLOWED_TABLES = new Set(['loyalty_clients', 'rewards', 'redemption_codes']);

export async function selectForUpdate(client: PoolClient, table: string, id: string) {
  if (!ALLOWED_TABLES.has(table)) throw new Error(`Table '${table}' not allowed`);
  const result = await client.query(
    `SELECT * FROM ${table} WHERE id = $1 FOR UPDATE`,
    [id],
  );
  return result.rows[0] || null;
}
```

### FINDING 10
**SEVERITY:** Medium  
**FILE:** `src/modules/clients/clients.repository.ts`  
**VULNERABILITY:** `findList` builds `sortColumn` by mapping `sort_by` query parameter to column expressions — if the allowlist mapping falls through to the default, it uses `lc.enrolled_at`. However the `sort_dir` parameter is mapped with `opts.sort_dir === 'asc' ? 'ASC' : 'DESC'` which is safe. The `sortColumn` mapping is safe as written, but the pattern of interpolating computed strings into ORDER BY is fragile. The `search` parameter is passed as `%${opts.search}%` to ILIKE — this is safe against injection but allows wildcard abuse (see A05).

No injection vulnerability as written, but the `sort_by` DTO uses `@IsIn([...])` validation which is the correct defense. **Confirmed safe — no finding.**

## A05 — Security Misconfiguration

### FINDING 11
**SEVERITY:** High  
**FILE:** `src/main.ts`  
**VULNERABILITY:** Security headers are applied via a raw middleware function but critical headers are missing: no `Content-Security-Policy`, no `Strict-Transport-Security`, no `Permissions-Policy`. The `X-XSS-Protection: 1; mode=block` header is deprecated and ineffective in modern browsers.

**FIX:**
```typescript
import helmet from 'helmet';
// npm install helmet

app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],
      styleSrc: ["'self'", "'unsafe-inline'"],
      imgSrc: ["'self'", 'data:', 'https:'],
    },
  },
  hsts: { maxAge: 31536000, includeSubDomains: true },
  referrerPolicy: { policy: 'strict-origin-when-cross-origin' },
}));
// Remove the manual middleware — helmet covers X-Content-Type-Options, X-Frame-Options
```

### FINDING 12
**SEVERITY:** High  
**FILE:** `docker-compose.yml`  
**VULNERABILITY:** `LOYALTY_JWT_SECRET: dev-secret-min-256-bits-change-in-production` and `INTERNAL_HMAC_SECRET: dev-hmac-secret-change-in-production` are hardcoded in `docker-compose.yml` which is committed to version control. If this file reaches a production-adjacent environment or CI/CD pipeline without override, these weak secrets are used.

**FIX:** Use Docker secrets or environment-specific `.env` files excluded from VCS:
```yaml
# docker-compose.yml — reference only, never set values
api:
  environment:
    LOYALTY_JWT_SECRET: ${LOYALTY_JWT_SECRET:?LOYALTY_JWT_SECRET must be set}
    INTERNAL_HMAC_SECRET: ${INTERNAL_HMAC_SECRET:?INTERNAL_HMAC_SECRET must be set}
```
Add `.env` to `.gitignore`. Use AWS Secrets Manager or SSM Parameter Store in production.

### FINDING 13
**SEVERITY:** Medium  
**FILE:** `src/modules/notifications/notifications.service.ts`  
**VULNERABILITY:** `getSmsUsage` accepts a `month` query parameter with no format validation — an attacker can pass arbitrary strings as the `month` parameter to `countSentThisMonth`, which constructs `${month}-01` and passes it to PostgreSQL. Malformed dates cause unhandled exceptions that leak stack traces via the global exception filter.

```typescript
// notifications.controller.ts
@Get('sms-usage')
async getSmsUsage(@Request() req: any, @Query('month') month?: string) {
  if (month && !/^\d{4}-\d{2}$/.test(month)) {

  }
  return this.service.getSmsUsage(req.user.location_id, month);
}
```

## A07 — Identification and Authentication Failures

### FINDING 14
**SEVERITY:** Critical  
**FILE:** `src/auth/pwa-jwt.guard.ts`  
**VULNERABILITY:** `PwaJwtGuard` reads the token from `request.query.t || request.params.token` but the `POST /redemptions/request` endpoint receives the token in the `Authorization: Bearer` header (as sent by `lib/loyalty-api.ts`). The guard never checks the Authorization header, so all PWA redemption requests receive `401 UnauthorizedException('Token required')`.

**FIX:**
```typescript
canActivate(context: ExecutionContext): boolean {
  const request = context.switchToHttp().getRequest();
  
  // 1. Authorization header (POST requests from PWA)
  const authHeader = request.headers['authorization'];
  let token = authHeader?.startsWith('Bearer ')
    ? authHeader.slice(7)
    : null;
  
  // 2. Query param (GET requests — SMS link clicks)
  if (!token) token = request.query.t ?? null;
  
  // 3. Never read from path params — tokens in URL paths appear in server logs
  if (!token) throw new UnauthorizedException('Token required');

  try {
    const secret = process.env.LOYALTY_JWT_SECRET;
    if (!secret) throw new Error('Secret not configured');
    const payload = jwt.verify(token, secret) as any;
    request.pwaClient = payload;
    return true;
  } catch (err: any) {
    if (err.name === 'TokenExpiredError') throw new GoneException('Link expired');
    throw new UnauthorizedException('Invalid token');
  }
}
```

### FINDING 15
**SEVERITY:** High  
**FILE:** `src/modules/pwa/pwa.controller.ts`  
**VULNERABILITY:** `GET /pwa/client/:token` exposes the JWT in the URL path — tokens appear in web server access logs, browser history, Referrer headers to third-party resources, and CDN/proxy logs. The security comment acknowledges this but the path-param route is still live.

```typescript
export class PwaController {

    if (!token) throw new UnauthorizedException('Token required');
    return this.service.getClientDashboard(token);
  }
}
```

### FINDING 16
**SEVERITY:** Medium  
**FILE:** `src/auth/hmac.guard.ts`  
**VULNERABILITY:** Replay window is 5 minutes (`Math.abs(now - requestTime) > 300`), which is reasonable, but the guard does not track used timestamps/nonces — the same request body can be replayed multiple times within the 5-minute window with the same valid HMAC and timestamp.

**FIX:** Store seen HMAC signatures in Redis with TTL matching the replay window:
```typescript
// In canActivate, after HMAC verification:
const nonce = `hmac-nonce:${expectedHmac}`;
const seen = await this.redis.get(nonce);
if (seen) throw new UnauthorizedException('Request already processed');
await this.redis.setEx(nonce, 300, '1');
return true;
```

## A08 — Software and Data Integrity Failures

### FINDING 17
**SEVERITY:** High  
**FILE:** `src/modules/internal/internal.service.ts`  
**VULNERABILITY:** `acknowledgeEventResult` trusts the worker's `dto.tier_updated` and `dto.new_tier_id` fields to update `current_tier_id` on the client record — a compromised worker or HMAC key leak allows an attacker to set any client's tier to any tier UUID, including tiers belonging to other locations.

**Evidence:**
```typescript
if (dto.tier_updated && dto.new_tier_id) {
  await this.pool.query(
    `UPDATE loyalty_clients SET current_tier_id = $1 WHERE id = $2`,
    [dto.new_tier_id, dto.loyalty_client_id], // no validation that tier belongs to client's location
  );
}
```

**FIX:** Recompute tier server-side from the authoritative balance, never trust the worker's tier assertion:
```typescript
// After recomputing computedBalance, recalculate tier internally
const tierResult = await this.pool.query(
  `SELECT lt.id FROM loyalty_tiers lt
   JOIN loyalty_configs lc ON lc.id = lt.loyalty_config_id
   JOIN loyalty_clients c ON c.location_id = lc.location_id
   WHERE c.id = $1
     AND $2 >= lt.min_points
     AND (lt.max_points IS NULL OR $2 <= lt.max_points)
   ORDER BY lt.min_points DESC LIMIT 1`,
  [dto.loyalty_client_id, computedBalance],
);
const newTierId = tierResult.rows[0]?.id ?? null;
// Update tier from server-computed value, not from worker payload
```

## A09 — Security Logging and Monitoring Failures

### FINDING 18
**SEVERITY:** Medium  
**FILE:** `src/common/filters/http-exception.filter.ts`  
**VULNERABILITY:** The exception filter logs `exception.message` and `exception.stack` for unhandled errors but logs nothing for `HttpException` instances — failed authentication attempts (401), authorization failures (403), and GDPR erasure requests (DELETE) produce no security audit trail beyond the HTTP response.

**FIX:**
```typescript
catch(exception: unknown, host: ArgumentsHost) {
  // ... existing status/message extraction ...
  
  // Security audit logging for sensitive status codes
  if (status === 401 || status === 403) {
    this.logger.warn(
      `[SECURITY] ${status} ${request.method} ${request.url} ` +
      `ip=${request.ip} user=${(request as any).user?.sub ?? 'anon'}`,
    );
  }
  if (status >= 500) {
    this.logger.error(`Unhandled error on ${request.method} ${request.url}`, 
      exception instanceof Error ? exception.stack : String(exception));
  }
  
  // Never include stack traces in API responses
  response.status(status).json({
    statusCode: status,
    message,
    ...(code ? { code } : {}),
    timestamp: new Date().toISOString(),
    path: request.url,
    // NO stack field
  });
}
```

## A10 — Server-Side Request Forgery (SSRF)

### FINDING 19
**SEVERITY:** High  
**FILE:** `src/modules/branding/branding.repository.ts` + `src/modules/rewards/dto/create-reward.dto.ts`  
**VULNERABILITY:** `logo_url` (branding) and `image_url` (rewards) accept arbitrary URLs validated only as syntactically valid URLs. When the PWA renders `<img src={logoUrl}>` or the worker calls `buildPwaLink`, no URL is fetched server-side here. However `BrandingEditor` and `RewardCard` render these URLs in `<img>` tags in the browser — a stored XSS/SSRF vector exists if an owner stores a `javascript:` URI or a URL pointing to an internal service (e.g., `http://169.254.169.254/latest/meta-data/` rendered as an image src can leak metadata via timing or error callbacks).

**FIX:**
```typescript
// hex-color.validator.ts — add URL allowlist validation
export function isValidHttpsUrl(url: string): boolean {
  try {
    const parsed = new URL(url);
    // Only allow https:// — no http, javascript, data URIs
    if (parsed.protocol !== 'https:') return false;
    // Block private IP ranges and metadata endpoints
    const hostname = parsed.hostname;
    if (/^(localhost|127\.|10\.|192\.168\.|169\.254\.|::1)/.test(hostname)) return false;
    return true;
  } catch { return false; }
}

// patch-branding.dto.ts
@IsOptional()
@IsString()
@Matches(/^https:\/\/.+/, { message: 'logo_url must be an HTTPS URL' })
logo_url?: string;
```
