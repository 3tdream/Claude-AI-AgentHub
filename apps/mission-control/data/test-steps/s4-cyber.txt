RISK LEVEL: Critical

Finding 1: PWA JWT Token Exposed in URL Path — Credential Harvesting via Logs, Referrer Headers, and Browser History
Risk: The `GET /api/v1/pwa/client/:token` endpoint embeds the signed `LOYALTY_JWT_SECRET`-validated token directly in the URL path, meaning it is written to every nginx/ALB access log, forwarded in `Referer` headers to any third-party resource loaded by the PWA (analytics, CDN, fonts), stored in browser history, and visible in server-side request logs across the entire infrastructure stack — any party with log access can replay the token to impersonate the client for the full `LOYALTY_LINK_EXPIRY_HOURS` window, accessing point balance, redemption codes, referral codes, and triggering reward redemptions via `POST /api/v1/loyalty/redemptions/request` which is auth'd only by this same PWA JWT.
Fix: Move the token to a query parameter (`/pwa/client?t=<token>`) as an immediate mitigation so it is excluded from standard path-based log formats, then implement a token-exchange flow where the SMS link carries a short-lived single-use opaque code that is exchanged server-side for a session cookie over HTTPS, eliminating long-lived bearer tokens in URLs entirely; simultaneously enforce `LOYALTY_LINK_EXPIRY_HOURS` to the shortest operationally acceptable value (recommend ≤ 1 hour) and add `Referrer-Policy: no-referrer` to all PWA responses.

Finding 2: Internal HMAC Endpoints Lack Network-Layer Isolation — Service-to-Service Auth Relies Solely on Application-Layer Secret
Risk: `POST /api/v1/internal/points-engine/event-result` and `POST /api/v1/internal/milestones/trigger` are protected only by an `INTERNAL_HMAC_SECRET` header validated in `src/auth/hmac.guard.ts`, but both endpoints share the same `/api/v1` base path and are deployed on the same CRM API surface exposed to the internet — if the HMAC secret is leaked (via `.env`, CI/CD logs, a misconfigured ECS task definition, or a path traversal in the app), an attacker can directly manipulate point balances for any client (`points_awarded`, `new_balance`, `tier_updated` fields are fully attacker-controlled in the event-result payload), trigger arbitrary milestone bonus awards, and poison the idempotency cache to suppress legitimate future events; additionally, the `event-result` endpoint accepts `new_balance` from the worker rather than recomputing it from the ledger, meaning the CRM trusts the worker's arithmetic — a compromised or spoofed worker can set any client's balance to an arbitrary integer.
Fix: Place internal endpoints behind a private VPC-internal ALB or security group rule that blocks all traffic not originating from the worker ECS task's security group, removing internet reachability entirely regardless of HMAC validity; refactor `POST /internal/points-engine/event-result` to discard the `new_balance` field from the request body and recompute the authoritative balance from the `PointTransaction` ledger server-side, treating the worker callback as a notification trigger only rather than a balance source of truth.

Finding 3: Redemption Code Enumeration and Timing Oracle on `POST /api/v1/loyalty/redemptions/:code/apply`
Risk: The apply endpoint returns distinct HTTP status codes for three different invalid-code states — `404` (code not found), `409` (code already used), and `410` (code expired) — and the `RedemptionCode.code` field is described as a "single-use alphanumeric" string of `max 20 chars` with no specified entropy floor or character set constraint in the data model or API contract, meaning an attacker with staff JWT credentials (a low-privilege role) can perform an online enumeration attack: a 404 response confirms the code does not exist, a 409 or 410 response confirms a valid code was issued and reveals its lifecycle state, leaking which clients have active or recently-used redemption codes; if codes are generated with insufficient entropy (e.g., 6-8 alphanumeric chars ≈ 36^8 ≈ 2.8 trillion but practically much less if a sequential or time-seeded PRNG is used), brute-force becomes feasible within the 60/min rate limit per the API contract, and a valid pending code can be applied by any staff member at any location without verifying the code belongs to a client at their location — the `location_id` scoping of the apply endpoint is not explicitly enforced in the API contract notes.
Fix: Enforce a minimum of 128 bits of cryptographic randomness for all redemption code generation using `crypto.randomBytes()` (not `Math.random()` or any PRNG), normalize all invalid-code responses to a single `404` status with a generic message to eliminate the timing/state oracle, add explicit `location_id` validation in `redemptions.service.ts` to reject apply attempts where the code's `location_id` does not match the staff JWT's location context, and reduce the apply endpoint rate limit from 60/min to 10/min with per-staff-id tracking rather than per-IP.
