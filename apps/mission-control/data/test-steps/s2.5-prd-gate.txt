### Criterion 3: Are scope boundaries clear (IN/OUT)?

**Assessment: YES — this is a strength of the PRD**

The IN/OUT/DEFERRED table is exceptionally well-structured. Observations:

- ✅ The three-table format (IN SCOPE / OUT OF SCOPE / DEFERRED with triggers) is unambiguous
- ✅ Deferred items include explicit trigger conditions (e.g., ">30% of beta users report external POS") — this prevents scope creep debates
- ✅ The HIGH-RISK defaults (Q1/Q2/Q3) are flagged with architectural implications, not just business ones
- ⚠️ **One boundary gap:** WhatsApp is listed as IN SCOPE ("WhatsApp as optional SMS alternative via Twilio WhatsApp channel") but there are **zero acceptance criteria** for WhatsApp behavior. Is it a complete feature or a configuration flag? AC-10 through AC-12 only reference SMS. If WhatsApp is truly in scope, it needs at minimum one AC covering opt-in, delivery failure handling, and message format parity with SMS. If it's just a config flag, that should be stated explicitly.

## FINAL VERDICT

```
GATE VERDICT: FAIL
```

**REASON:** The PRD is exceptionally well-structured and passes on scope clarity, AC format rigor, and risk documentation — it is above average quality for a V1 draft. However, two blocking ambiguities prevent the Architect from proceeding without guessing on critical integration points: (1) the checkout event contract (payload schema, producer/consumer pattern) that drives the entire points engine is undefined, despite the env var referencing a queue; and (2) the referral code capture mechanism at booking time has no specified integration point, making it impossible to design the referral attribution flow. Additionally, WhatsApp is listed as IN SCOPE but has zero acceptance criteria, creating a scope boundary inconsistency. These gaps are surgical — the PRD does not need structural revision, only targeted additions to three specific areas.

## MISSING ITEMS (Required for PASS)

**🔴 BLOCKING — Must be added before Architect proceeds:**

1. **[EVENT CONTRACT]** Add a section or appendix defining the checkout event schema that the points engine consumes. Minimum required fields: `event_type`, `client_id`, `location_id`, `organization_id`, `transaction_amount`, `service_line_items[]`, `payment_confirmed_at`, `staff_id`. Specify whether this is synchronous (direct call) or asynchronous (queue message). This resolves G1 and unblocks the entire points engine design.

2. **[REFERRAL CAPTURE AC]** Add AC-18b (or amend US-06 scope): specify *how* a referral code is associated with a new client's first booking — e.g., "GIVEN a new client arrives via a referral URL containing `?ref=CODE`, WHEN they complete enrollment, THEN the referral code is stored against their profile and evaluated at first-visit checkout completion." Without this, the booking→referral attribution integration point is undesigned.

**🟡 NON-BLOCKING — Should be added before Sprint 1 kickoff:**

3. **[WHATSAPP AC]** Either add at least one acceptance criterion for WhatsApp behavior (opt-in capture, delivery failure handling, message content parity with SMS) — or explicitly reclassify WhatsApp from IN SCOPE to DEFERRED with a feature flag note. The current state is a scope boundary inconsistency.

4. **[BIRTHDAY SCHEDULER SPEC]** Amend AC-19 to specify time-of-day for birthday SMS delivery (e.g., "bonus points are awarded and SMS sent at 10:00 AM in the salon's configured timezone"). This has direct TCPA compliance implications and prevents an Architect from making an arbitrary scheduling decision.

5. **[PWA RE-AUTH FLOW]** Add one AC or a note in US-05 scope covering the expired-link re-authentication path for returning PWA clients (e.g., phone number re-entry → new SMS link). This is a security boundary decision, not just a UX one.
