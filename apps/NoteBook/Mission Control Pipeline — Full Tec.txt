Mission Control Pipeline — Full Technical Report

  Architecture

  User Input → Smart Router → [Route Decision] → Pipeline Executor → [13 Stages] → Result
                                                        ↓
                                                   Per-Stage:
                                                   Cache Check → Execute Agent → Evaluate → Retry/Pass/Escalate
                                                        ↓
                                                   Learning DB ← save results

  Execution Flow

  1. User writes task in /orchestration input
  2. Smart Router analyzes → picks mode (quick/medium/full) + agents
  3. User can override mode before Execute
  4. Pipeline Executor runs stages sequentially:

     For each stage:
     ├─ Check Task Cache → if cached output exists (score ≥ threshold) → SKIP (reuse)
     ├─ Check Pause/Stop flags → if set → halt
     ├─ Check if resumed → if completed in previousExecution → SKIP
     ├─ Inject learning context (agent stats + recent runs)
     ├─ Build prompt (template + upstream outputs + project context + tool instructions)
     ├─ Call /api/ai/execute (with or without tools)
     │   ├─ Without tools: single LLM call → text response
     │   └─ With tools: multi-turn loop (stream mode):
     │       ├─ LLM responds with tool_use blocks
     │       ├─ Execute tools server-side (read_file, edit_file, etc.)
     │       ├─ Truncate tool output (read: 12K, cmd: 10K, other: 8K)
     │       ├─ Check read budget → inject warning if exceeded
     │       ├─ Send results back to LLM
     │       └─ Repeat until LLM stops or maxTurns reached
     ├─ Evaluate output:
     │   ├─ Auto-fail if < 100 chars
     │   ├─ Send to Orchestrator LLM (sonnet) for 4-axis scoring
     │   ├─ Parse: [SCORE] completeness, specificity, actionability, taskCompletion → PASS/FAIL
     │   ├─ Calculate weighted overall (per-agent weights from AGENT_SCORING_WEIGHTS)
     │   └─ Override PASS/FAIL based on math (overall ≥ threshold)
     ├─ If PASS → save to context, move to next stage
     ├─ If FAIL:
     │   ├─ Smart retry: stop if score degraded (< lastScore - 0.1)
     │   ├─ If worth retrying → surgical retry prompt (clean context, exact feedback)
     │   ├─ If max retries (3) reached:
     │   │   ├─ Score ≥ escalationThreshold → accept with warning
     │   │   └─ Score < escalationThreshold → ESCALATE (halt pipeline)
     │   └─ Update lastScore tracker
     └─ Save to analytics (pipeline-runs/, pipeline-analytics.json)

  5. After all stages: save full run to learning DB
  6. Fire-and-forget: Jira sync, knowledge enrichment

  Scoring System

  4 axes, role-aware, per-agent weights:

  overall = taskCompletion × W_task + completeness × W_comp + specificity × W_spec + actionability × W_act

  ┌────────────────┬───────────────────────────┬────────────────────────────────────────────────────────────────────┐
  │      Axis      │           What            │                           Role examples                            │
  ├────────────────┼───────────────────────────┼────────────────────────────────────────────────────────────────────┤
  │ TaskCompletion │ Did agent DELIVER?        │ Backend: edit_file succeeded. PM: PRD delivered. QA: VERDICT given │
  ├────────────────┼───────────────────────────┼────────────────────────────────────────────────────────────────────┤
  │ Completeness   │ All requirements covered? │ Architect: referenced all files. PM: all acceptance criteria       │
  ├────────────────┼───────────────────────────┼────────────────────────────────────────────────────────────────────┤
  │ Specificity    │ Concrete paths, names?    │ Backend: exact endpoints. Architect: exact FILES_TO_EDIT           │
  ├────────────────┼───────────────────────────┼────────────────────────────────────────────────────────────────────┤
  │ Actionability  │ Next agent can work?      │ Architect→Backend: detailed instructions. QA: repro steps          │
  └────────────────┴───────────────────────────┴────────────────────────────────────────────────────────────────────┘

  Evaluator: Orchestrator (sonnet) scores each output. Sees first 4000 chars of agent output.

  Retry Protocol

  Attempt 1 → score X → lastScore = X
    If X ≥ threshold → PASS
    If X < threshold → retry (always retry once)

  Attempt 2 → score Y → lastScore = Y
    If Y ≥ threshold → PASS
    If Y < lastScore - 0.1 → DEGRADED → STOP (accept with warning or escalate)
    If Y > lastScore + 0.5 → improving → retry again

  Attempt 3 (max) →
    If score ≥ escalationThreshold → accept with warning
    If score < escalationThreshold → ESCALATE → pipeline paused

  Retry prompt: clean context (no previous output), exact failure reason, score shown.

  Backend special rule: First tool call MUST be edit_file or create_file. read_file only if needed for exact old_string.

  Read budget warning: After N reads (per-agent config), message injected: "STOP READING, ACT NOW".

  Tool output truncation: Appended to last tool_result content (not separate message, avoids API error).

  Caching & Resume

  Task Cache:
  Before each stage → GET /api/pipeline/cache?input=X&stepId=Y&minScore=Z
  If match found → reuse output, mark as "cached", skip execution
  Saves ~100K tokens on repeated tasks

  Resume:
  executePipeline(steps, input, ..., previousExecution)
  → Completed stages from previousExecution → skip, reuse outputs
  → Failed/pending stages → re-execute
  → UI: "Resume" button on failed/paused/stopped executions

  Pause/Stop:
  Pause → pauseRequested flag → current stage finishes, next doesn't start
  Stop → stopRequested flag → halt between stages
  UI: Pause/Stop buttons during running execution

  1. Backend stuck — reads files but never edits. "Edit first" prompt + 3 max turns is latest fix (untested)
  2. PM borderline — scores 6.9-7.4, retries sometimes degrade
  3. Full mode never completed — always fails at Backend (s4)
  4. No full step outputs in older runs — only runs after stepOutputs feature have cached data
  5. Agent Hub legacy code — unused but still in project, causes confusion
